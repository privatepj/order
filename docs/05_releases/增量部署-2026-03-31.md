## 0. 增量脚本规范（重要）

- **除全量 schema**（如 `scripts/sql/00_full_schema.sql`、`app/scripts/sql/00_full_schema.sql`）外，`scripts/sql/run_*.sql` **一律不得对已存在的文件做就地修改**，只能**新增**下一个 `run_NN_*.sql`。
- 仓库规则见：`.cursor/rules/incremental-sql-append-only.mdc`。
- **run_31 说明**：`run_31_production_rbac.sql` 保持历史已发布语义（`production` 挂在 `parent_id=2` 下，`sort_order=50`）。曾误并入 run_31 的「顶级菜单」修正已拆到 **`run_38_production_rbac_nav_top_level.sql`**（与 **`run_32_production_nav_root.sql`** 语义相同；若已按顺序执行过 run_32，run_38 可跳过，重复执行亦幂等）。

## 2. 增量 SQL 执行顺序（必须按顺序）

数据库：`sydixon_order`

按以下顺序执行：
1. `scripts/sql/run_31_production_rbac.sql`
2. `scripts/sql/run_32_production_nav_root.sql`
3. `scripts/sql/run_33_hr_module.sql`
4. `scripts/sql/run_34_machine_module.sql`
5. `scripts/sql/run_35_procurement_module.sql`
6. `scripts/sql/run_36_production_incident.sql`
7. `scripts/sql/run_37_express_waybill_batch_delete.sql`

说明：
- 所有脚本遵循“无外键约束”规则（仅逻辑关联 + 索引）。
- `run_33/34/35` 含 `DROP TABLE IF EXISTS` + `CREATE TABLE`，对目标表是重建行为，执行前请确认环境是否允许覆盖旧数据。
- `run_32/36/37/38` 影响导航/能力/角色授权，执行后需刷新 RBAC 缓存（重启应用或调用缓存失效逻辑）。

## 5. OpenClaw 上机部署步骤（建议）

## 5.1 部署前检查
- 确认当前分支代码已包含上述文件（尤其 `routes_*`、`models/*`、`templates/*`、`scripts/sql/run_31~38`）。
- 备份 `sydixon_order`（至少备份受影响表和 RBAC 表）。
- 确认部署环境 Python 依赖无额外新增（本次未引入新第三方包）。

## 5.2 执行数据库脚本
- 按第 2 节顺序执行 `run_31` 到 `run_38`（run_38 若与 run_32 重复可按第 0 节跳过）。
- 若采用批量执行，确保中途失败即停止，不继续后续脚本。

## 5.3 发布应用代码
- 同步代码到目标机。
- 重启 Web 进程（建议完整重启，确保菜单/能力缓存生效）。

## 5.4 缓存处理
- 若有显式 RBAC 缓存失效入口，可调用 `invalidate_rbac_cache()`。
- 若无，直接重启进程是最稳妥方式。
