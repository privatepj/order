# 增量部署说明（2026-03-28）

本文档对应**本次发布批次**的代码与 `scripts/sql` 变更，用于**已有数据的生产库**安全升级。执行任何数据库脚本前请先**全库备份**（`mysqldump` 或快照）。

---

## 1. 本次发布内容范围

| 类别 | 说明 |
|------|------|
| 应用代码 | `app/`（路由、模型、服务、模板、OpenClaw 等） |
| 环境变量示例 | `.env.example`（如 `INVENTORY_DEFAULT_STORAGE_AREA`、`OPENCLAW_API_KEY` 等） |
| SQL | `scripts/sql/` 下 `01_migrations_for_old_db.sql` 与各 `run_*.sql` |
| 辅助脚本 | `scripts/run_inventory_migration.py`、`scripts/run_mcp_sql_migration.py`（可选，本地/运维用） |

**勿将** `.env`、含密码的 `.cursor/mcp.json` 提交仓库或原样拷到公开环境。

---

## 2. 数据库红线（避免误删）

| 脚本 | 用途 | 生产已有数据时 |
|------|------|----------------|
| `scripts/sql/00_full_schema.sql` | 全量建表 + 初始数据 | **禁止执行**。内含 `DROP TABLE IF EXISTS`，会清空业务表与数据。 |
| `scripts/sql/01_migrations_for_old_db.sql` | 从旧库合并的增量迁移 | **仅适用于尚未合并过该批次的旧库**；若已执行过，不要整文件重跑（大量 `ALTER ADD` 会报重复）。 |
| `scripts/sql/run_*.sql` | 单项补丁 | 按实际上线库缺失项**按需、按序**补跑。 |

**新服务器、空库**：只执行 `00_full_schema.sql`（见 `scripts/sql/README.md`）。

---

## 3. 增量 SQL 执行策略

### 3.1 从未跑过合并迁移的旧库

1. 备份数据库。
2. 执行：

   ```bash
   mysql -u 用户名 -p sydixon_order < scripts/sql/01_migrations_for_old_db.sql
   ```

3. 若某条报错为「Duplicate column」「Table already exists」等，表示该步已做过，**跳过该条**继续后续（或分段执行）。

### 3.2 已跑过新版 `01`，仅需补当前应用依赖

按**缺什么补什么**（均在 `scripts/sql/` 下），执行前已备份：

| 顺序建议 | 文件 | 说明 |
|----------|------|------|
| 1 | `run_16_seed_nav_capability.sql` | 导航与能力**基线**种子；含 `ON DUPLICATE KEY UPDATE`，可重复执行。**勿再向本文件追加新能力键**（已执行过旧版 run_16 的库不会重跑整文件逻辑上也不会补新行）；新能力只用下面单独 `run_XX`。 |
| 2 | `run_17_migrate_role_json_to_nav_cap.sql` | 将 `role.allowed_menu_keys` / `allowed_capability_keys` 写入 `role_allowed_nav` / `role_allowed_capability`（`INSERT IGNORE`）。**不要**取消注释其中的 `TRUNCATE`，否则会清空关联表。 |
| 3 | `run_18_add_inventory_movement_list_cap.sql` | 补能力键 `inventory_ops.movement.list`（幂等）。 |
| 4 | `run_20_delivery_edit_delivery_no_cap.sql` | 补能力键 `delivery.action.edit_delivery_no`（幂等）。 |
| 5 | `run_21_delivery_edit_waybill_cap.sql` | 补能力键 `delivery.action.edit_waybill`（幂等）；可选同步角色见脚本内注释。 |
| 6 | `run_19_delivery_express_nullable.sql` | 若 `delivery.express_company_id` 仍需允许自配送为 `NULL` 而尚未 `MODIFY`，再执行（新版 `01` 末尾已含同类语句，可能已做过）。 |

**通常不必再执行**（与 `01` 或彼此重复）：

- `run_13_inventory_ledger.sql`：期初/进出表已在 `01` 中 `CREATE TABLE IF NOT EXISTS`。
- `run_14_role_capability_keys.sql`：`allowed_capability_keys` 列已在 `01` 中 `ALTER`。
- `run_15_nav_rbac_schema.sql`：与 `01` 中导航/RBAC 建表段落重复。

**条件执行**：

- `run_drop_product_model.sql`：仅当确认 `product` 表仍存在 `model` 列且业务不再需要时执行；列已不存在可忽略报错。

### 3.3 部署后快速自检（MySQL）

SHOW TABLES LIKE 'role_allowed_nav';
-- 可选：查看 delivery 表 express_company_id 是否允许 NULL

## 4. 应用与配置发布

1. 发布代码到服务器（Git pull 或等价方式）。
2. 激活虚拟环境后：`pip install -r requirements.txt`（依赖有变更时）。
3. 对照 `.env.example` 更新生产环境 `.env`：
   - 必填：`DATABASE_URL`、`SECRET_KEY`
   - 送货自动出库：`INVENTORY_DEFAULT_STORAGE_AREA`
   - OpenClaw 接口：`OPENCLAW_API_KEY`
4. 重启应用进程（如 gunicorn / systemd）。完整环境要求见根目录《项目整体部署.md》。
